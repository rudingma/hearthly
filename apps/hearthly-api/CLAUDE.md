# Hearthly API — Project Instructions

## API Endpoints

- **No global prefix** — `api.hearthly.dev` subdomain handles routing, no `/api` prefix
- **Health:** `GET /health` (Terminus, no DB health indicators — always 200 if process is alive, intentional)
- **GraphQL:** `POST /graphql` (Apollo Server 5, code-first, introspection enabled in non-prod)
- `@as-integrations/express5` must be in both root and app `package.json` for workspace hoisting

## GraphQL (Code-First)

- Code-first: TypeScript decorators (`@ObjectType`, `@Field`, `@Query`) generate schema at startup
- **Resolver → Service → Repository → Drizzle** — resolvers are thin, services are framework-agnostic
- **Depth limit 7, query complexity limit 1000** (configured in `src/app/app.module.ts`). `graphql-depth-limit` has a custom type declaration at `src/types/graphql-depth-limit.d.ts`.
- Schema file `src/schema.gql` is auto-generated by `npx nx generate-schema hearthly-api`. When adding new resolvers, register them in `src/generate-schema.ts`. Do not edit `schema.gql` by hand.
- Design doc: `docs/api-design.md`

## Module Structure

Root module: `src/app/app.module.ts`. Each domain module lives under `src/modules/<name>/`:

- `<name>.module.ts` — NestJS module
- `<name>.service.ts` — Business logic (framework-agnostic)
- `<name>.repository.ts` — Drizzle queries via `TransactionHost`
- `models/<name>.model.ts` — `@ObjectType()` GraphQL type
- `resolvers/<name>.resolver.ts` — `@Query()` and `@Mutation()`
- `schema/<name>.table.ts` — Drizzle `pgTable` definition
- `schema/index.ts` — Barrel re-export

**Key conventions:**

- Services return plain TypeScript types, NOT GraphQL types
- Repositories use `TransactionHost` (never direct `@Inject(DRIZZLE)`). `TransactionHost` requires a CLS context — do not call repository methods from cron jobs or startup code without wrapping in `ClsService.run()`.
- New schemas must be re-exported from `src/database/schema.ts`
- Auth module exports via barrel `src/modules/auth/index.ts`: `Public`, `CurrentUser`, `JwtPayload`. Import from the barrel, not individual files.
- Reference: `docs/data-layer-design.md`

## Drizzle Migrations

Run from `apps/hearthly-api/` (not repo root):

1. Change schema in `schema/<name>.table.ts`
2. Run `npx drizzle-kit generate` to create migration
3. Run `npx drizzle-kit migrate` to apply locally
4. Commit the generated migration file with your code change
5. Migrations auto-apply on API startup in production

**Column conventions:** `text` over `varchar`, `timestamptz` over `timestamp`

## Configuration

- **Env validation uses Zod** (not class-validator). Config split into `appConfig`, `databaseConfig`, `authConfig` via `registerAs()` in `src/config/`. Inject via `@Inject(appConfig.KEY)` with the corresponding type.
- **Build uses webpack** (see `webpack.config.js`). Migrations bundled via `assets` config. Output is a single JS bundle.

## Authentication (OIDC / Keycloak)

- **Library:** `jose` (JWT verification, JWKS fetching — NOT Passport.js)
- **Guard:** `JwtAuthGuard` — global via `APP_GUARD`, verifies JWT signature/issuer/audience
- **Decorators:** `@Public()` opts out of auth. `@CurrentUser()` extracts `JwtPayload` from context — **GraphQL-only**, returns `undefined` in HTTP contexts.
- **Guard has no DB access** — only verifies tokens. User sync (`getOrSyncByKeycloakId`) happens in resolvers.
- **`JwtPayload`:** `{ sub, email, name?, picture?, roles }` — token claims, not a DB entity
- **Config:** `KEYCLOAK_ISSUER_URL` and `KEYCLOAK_CLIENT_ID` via `@nestjs/config` `ConfigService`

### Local Keycloak

- `docker compose up -d` starts Keycloak on `http://localhost:8180`
- Test user: `dev@hearthly.dev` / `dev` (10-hour access tokens)
- Get token: `curl -s -X POST http://localhost:8180/realms/hearthly/protocol/openid-connect/token -d "grant_type=password&client_id=hearthly-app&username=dev@hearthly.dev&password=dev"`

## Environment Variables

Copy `.env.example` to `.env` at the monorepo root. Key variables:

- `DATABASE_URL` — local PostgreSQL (port 5434, not default 5432)
- `KEYCLOAK_ISSUER_URL` — local Keycloak realm URL
- `KEYCLOAK_CLIENT_ID` — OIDC client ID (`hearthly-app`)
- `CORS_ORIGIN` — frontend origin (`http://localhost:4200`)
- `GOOGLE_CLIENT_SECRET` — optional for local dev (Keycloak import uses placeholder fallback)

## Testing

- **Framework:** Vitest (config at `vitest.config.ts`, run from `apps/hearthly-api/`)
- **Unit tests:** `*.spec.ts` — mock repository with `vi.fn()`, wire via `@nestjs/testing`
- **Integration tests:** `*.integration.spec.ts` — PGlite (in-memory Postgres), shared helper at `test/support/test-db.ts`
- **Test JWTs:** `test/support/test-jwt.ts` — `generateTestKeyPair()`, `createTestJwks()`, `signTestToken()` for auth tests without a real Keycloak
- **Pattern guide:** `docs/data-layer-design.md` Section 4

## Known Issues

- **Bun runtime:** Production image uses `oven/bun:1.2.20-alpine`. NestJS is not officially supported on Bun — upgrade Bun versions intentionally after 4-6 weeks of community validation, not automatically.
